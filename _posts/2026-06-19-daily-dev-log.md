## Today's Update

Three distinct bodies of work closed out today, and the through-line between them is probably "enforcing discipline on things that are easy to defer." The governance receipts feature went from boundary ADR to full archive. The TODO remediation initiative reached its own completion state. And the unified publishing content library got built end-to-end, including frontend, API, and docs.

Governance receipts was the largest sustained effort. The scope here is straightforward to describe and genuinely complex to execute: when the platform makes a governed decision — via TrustFabric's `PolicyDecisionRecorded` event, an MCP execution, or a compliance publication — the system needs to produce an auditable receipt, route it to the right audience with the right level of detail, and deliver it through whatever channels the delivery policy specifies. I started with the boundary ADR and schema design, which forced early decisions about the four core tables: `governance_receipts`, `governance_receipt_delivery_policies`, `governance_receipt_deliveries`, and `governance_receipt_acknowledgements`. The separation between the receipt itself and its delivery records is load-bearing — it lets the system retry deliveries independently without touching the canonical receipt record. From there I built the full Porto stack: models and factories for all four tables, the `GovernanceReceiptSourceEvent` DTO, event listeners for the three source events, the audience profile enum and field policy that controls what data is safe to surface per recipient type, the `GenerateGovernanceReceiptAction`, delivery tasks for inbox, email, and webhook channels, and the delivery dispatch job that orchestrates all three. The audience-safe renderer is the piece I'm least settled on — it works correctly against the tests, but the field policy mechanism for controlling which receipt fields are visible to which audience profiles is going to need revisiting if the audience taxonomy expands significantly. Right now it's a clean enum-driven approach; at higher cardinality it may need something more like a policy table. The API surface covers list, detail, acknowledge, archive, export, and delivery policy management endpoints, all gated by the authorization policy. Frontend built out as a `My Receipts` list and detail with accessibility tests alongside unit tests.

The TODO remediation initiative closed out at the same time, which was mostly execution rather than design. The meaningful technical work was the `TodoFormatRule` activation — adding `todo-format-rules.neon` to the PHPStan config, verifying the pre-commit hook picks it up, and confirming CI actually fails on a deliberately malformed TODO. That last check matters: it's easy to have a rule that passes linting but doesn't block the commit path, and those rules accumulate into false confidence. The stale TODO triage was genuinely useful — `SyncCrmContactsAction.php` had a placeholder ID generation comment that had been sitting there long enough to be archaeologically interesting, and the ETL private routes provider-alias cluster had three separate TODOs that all pointed at the same deferred cleanup. I resolved the live ones, registered the deferred ones in `DEFERRED-BACKLOG.md`, and archived the planning directory. The `Telemetry/DatabaseQueryListener.php` TODO was the only one in the batch that needed actual implementation rather than reformatting or deferral — that one got implemented.

The unified publishing content library rounds out the day. The design problem here is that publication content exists in two shapes simultaneously: page records in Core Publishing and post snapshots in the AppSection Publication container. A content manager navigating the admin shouldn't have to know that distinction exists. The `ListPublicationPostContentItemsAction` and corresponding page action both project their source data into a shared `ContentItemResponseData` shape, and the `ComposeContentItemProjectionAction` merges them. The `GET /api/content-items` route returns that unified view, gated by the read-only `PublishingContentReader` authorization policy. On the frontend, the panel layout consumes that endpoint and surfaces URL evidence, placements, and embedded form usage summaries in one place, with filter support. I wrote tests at each layer — tenant filtering for post snapshots, filter behavior, page and post mapping fidelity, the unified API response shape, and frontend service parsing. The action links from the content library panel back to existing detail surfaces were the last piece, and they depend on the URL evidence rendering being correct, so the ordering mattered there.

Planning documentation and a handful of admin surface standardization items also cleared, though those were more process than architecture. Tomorrow the active surface is the new planning entries that were added today: admin UI improvements, a search filter pass, publication identity work, and an admin routes refactor. The receipts and content library work being complete and archived means those don't carry any planning overhead into next week.
